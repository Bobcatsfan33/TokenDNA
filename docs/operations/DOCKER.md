# Container image — `ghcr.io/bobcatsfan33/tokendna`

Multi-arch (linux/amd64 + linux/arm64) distroless image of the TokenDNA backend. Published to GitHub Container Registry on every `v*` tag by `.github/workflows/release-docker.yml`. Signed with **cosign keyless OIDC** so anyone can verify the image came from this repo's release workflow without trusting a long-lived PGP key.

## Quick start

```bash
docker run --rm -p 8000:8000 \
  -e DEV_MODE=true \
  -e ATTESTATION_CA_SECRET=demo-secret-32-bytes-aaaaaaa \
  ghcr.io/bobcatsfan33/tokendna:latest
```

The container exposes port 8000. Once it's up:

```bash
curl http://127.0.0.1:8000/api/health
# → {"redis": ..., "clickhouse": ..., "tor_list": {"ok": true ...}, ...}
```

### Persisting the SQLite database

The container runs as the distroless `nonroot` user (UID 65532) and writes the dev SQLite DB to `/data/tokendna.db`. To keep that DB across restarts, mount a host **directory** (not just the file) so SQLite can also create its `-wal` and `-shm` sidecar files:

```bash
mkdir -p /tmp/tokendna-data
chmod 777 /tmp/tokendna-data        # writable by container's UID 65532

# (optional) seed it from the dev fixture before first boot
cp /tmp/tokendna-demo.db /tmp/tokendna-data/tokendna.db
chmod 666 /tmp/tokendna-data/tokendna.db

docker run --rm -p 8000:8000 \
  -e DEV_MODE=true \
  -e ATTESTATION_CA_SECRET=demo-secret-32-bytes-aaaaaaa \
  -e DEV_TENANT_ID=acme \
  -v /tmp/tokendna-data:/data \
  ghcr.io/bobcatsfan33/tokendna:latest
```

For production, swap SQLite for Postgres via `DATABASE_URL` and you don't need a host volume at all.

## Tags

Tags are written by `docker/metadata-action` from each `v*` tag push:

| Tag pattern                  | Example         | Use case                                              |
|------------------------------|-----------------|-------------------------------------------------------|
| `vX.Y.Z`                     | `v0.2.3`        | **Always pin to this in production manifests**        |
| `X.Y.Z`                      | `0.2.3`         | Same image, no `v` prefix                             |
| `X.Y`                        | `0.2`           | Stays on the latest patch within a minor             |
| `X`                          | `0`             | Stays on the latest minor.patch within a major       |
| `latest`                     | `latest`        | Cutting edge of the default branch — dev only         |
| `edge`                       | `edge`          | Manual `workflow_dispatch` build (off-cycle smoke)    |

All tags for one release share a single image digest, so `cosign verify` against any of them succeeds — but you should pin by digest in any reproducible deployment:

```bash
docker pull ghcr.io/bobcatsfan33/tokendna:v0.2.3
DIGEST=$(docker inspect --format '{{index .RepoDigests 0}}' ghcr.io/bobcatsfan33/tokendna:v0.2.3)
echo "$DIGEST"
# → ghcr.io/bobcatsfan33/tokendna@sha256:<64-char-digest>
```

Use the digest form in your Helm chart / k8s manifests / docker-compose.

## Verifying the supply chain

The release workflow signs every published manifest with cosign in keyless mode (no key material to manage; the proof is the GitHub OIDC identity at sign time). Anyone, anywhere can verify before pulling:

```bash
cosign verify ghcr.io/bobcatsfan33/tokendna:v0.2.3 \
  --certificate-identity-regexp 'https://github.com/Bobcatsfan33/TokenDNA/.github/workflows/release-docker.yml@.*' \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com
```

A successful verification means: the image was built and pushed by *this repo's* `release-docker.yml` workflow, on *this repo's* GitHub Actions runners, in response to a tag push that you can find in the Actions tab. **No long-lived signing key exists** — there is nothing to leak.

The build also attaches:
- An **SLSA build-provenance attestation** (`provenance: true` in the workflow).
- An **SPDX SBOM attestation** (`sbom: true` — generated by Buildx).

Pull them like any other cosign attestation:

```bash
cosign download attestation ghcr.io/bobcatsfan33/tokendna:v0.2.3 \
  --predicate-type https://spdx.dev/Document
```

## Configuration via environment variables

The image runs `uvicorn api:app` — every env var the API supports works inside the container. The minimum production set:

| Env var                                              | Notes                                                         |
|------------------------------------------------------|---------------------------------------------------------------|
| `TOKENDNA_ENV`                                       | `production` enables the secret-gate fail-closed behaviour    |
| `ATTESTATION_CA_SECRET` _or_ `ATTESTATION_KEY_BACKEND=aws_kms` + `AWS_KMS_KEY_ID` | Trust authority signing material |
| `TOKENDNA_DELEGATION_SECRET` / `_WORKFLOW_SECRET` / `_HONEYPOT_SECRET` / `_POSTURE_SECRET` | Module-specific HMAC secrets   |
| `DATA_DB_PATH`                                       | SQLite path; for production set up Postgres via `DATABASE_URL` |
| `REDIS_HOST` + `REDIS_TLS=true`                      | Redis with mTLS                                               |
| `CLICKHOUSE_HOST` + `CLICKHOUSE_SECURE=true`         | ClickHouse with TLS                                           |
| `TLS_CA_CERT_PATH`, `TLS_API_CERT_PATH`, `TLS_API_KEY_PATH` | Internal mTLS — see `docs/operations/MTLS.md`           |
| `EDGE_SYNC_TOKEN`                                    | Cloudflare Worker snapshot endpoint auth                      |
| `FIELD_CRYPTO_KEY` _or_ `FIELD_CRYPTO_KEYRING`       | At-rest column encryption                                     |
| `DEV_MODE`                                           | `true` only for local evaluation                              |

Mount cert/secret files via Docker volumes (read-only):

```bash
docker run --rm -p 8000:8000 \
  -v $PWD/deploy/tls:/etc/tokendna/tls:ro \
  -e TLS_CA_CERT_PATH=/etc/tokendna/tls/ca.crt \
  -e TLS_API_CERT_PATH=/etc/tokendna/tls/api.crt \
  -e TLS_API_KEY_PATH=/etc/tokendna/tls/api.key \
  -e TOKENDNA_ENV=production \
  --env-file ./prod.env \
  ghcr.io/bobcatsfan33/tokendna:v0.2.3
```

## docker-compose

The repo's `docker-compose.production.yml` already references the local build target. To use the published image instead, set the `image:` directive:

```yaml
services:
  tokendna:
    image: ghcr.io/bobcatsfan33/tokendna:v0.2.3
    # remove the `build:` block if you previously had one
```

## Air-gapped delivery

Federal customers operating sealed networks can pull the image to a connected jumpbox once, save it as a tarball, and `docker load` it inside the boundary:

```bash
# Connected machine
docker pull ghcr.io/bobcatsfan33/tokendna:v0.2.3
docker save ghcr.io/bobcatsfan33/tokendna:v0.2.3 -o tokendna-v0.2.3.tar
# Transfer tokendna-v0.2.3.tar via your approved data-diode / removable media

# Air-gapped machine
docker load -i tokendna-v0.2.3.tar
```

`cosign verify` over the offline copy still works as long as the cosign binary and the Sigstore root certificate (Fulcio + Rekor public keys) are pre-staged on the air-gapped side. Have your acquisition lead pull the Sigstore root once and stage it alongside the image tarball.

## CI smoke

The release workflow includes a self-verify step that runs `cosign verify` against the just-pushed digest before declaring success. If signing or verification fails, the publish step exits non-zero and the GitHub release page shows it as a failed build — there is no half-pushed manifest.

## What this changes for adoption

- **Evaluation onboarding** drops from "clone, install Python, configure secrets" to one `docker run` line.
- **Air-gapped + IL5/IL6 customers** get a tarball-portable artefact instead of a Git URL.
- **Helm chart** in `deploy/helm/` becomes deployable verbatim — replace `image: tokendna` with the GHCR coordinates.
- **Compliance posture** improves: signed images, SBOM attestation, SLSA provenance — all of which appear in `compliance.generate_evidence_package` output for SC-12, SR-3, SR-4 controls.
