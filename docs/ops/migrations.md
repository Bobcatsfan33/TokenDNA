Alembic is the schema-change pipeline for TokenDNA. The repository owns:

* `alembic.ini` â€” Alembic config (URL resolved from env, see below).
* `alembic/env.py` â€” environment hook that resolves the SQLAlchemy URL

## URL resolution

`alembic/env.py` resolves the database URL in this priority order:

1. `TOKENDNA_PG_DSN` â€” production / staging Postgres DSN (the same env
   var the runtime uses).
2. `TOKENDNA_ALEMBIC_URL` â€” explicit override (used in tests / dry runs).
3. `sqlalchemy.url` in `alembic.ini` â€” dev fallback (`sqlite:///./local.db`).

## Common operations

### Apply all pending migrations

```bash
TOKENDNA_PG_DSN="postgresql://tokendna:***@db:5432/tokendna" \
  alembic upgrade head
```

### Stamp an existing database with the baseline

If a tenant already runs the schema and you only need Alembic to know
the current revision:

```bash
alembic stamp 0001_baseline
```

### Generate a new revision

```bash
alembic revision -m "add idx_passport_subject"
# edit alembic/versions/<rev>_add_idx_passport_subject.py
alembic upgrade head
```

We do not use `--autogenerate`. TokenDNA's schema lives as raw DDL in
each module, not as SQLAlchemy declarative models, so autogenerated
migrations would not see the table set.

## Baseline philosophy

`0001_baseline` walks every TokenDNA module that owns its own schema and
calls its `init_db()` against the Alembic-managed connection. This keeps
a single source of truth for table definitions:

* Application code uses `init_db()` at startup (or the runtime
  preflight in `_startup_checks`).
* Alembic invokes the same `init_db()` against the migration connection.

If you split a module's DDL across files, keep `init_db()` as the public
entry point so this contract holds.

## Downgrade policy

`0001_baseline.downgrade()` raises `NotImplementedError`. Production
databases must not be torn down by a single Alembic command. To rebuild
from scratch, drop the schema externally and run `alembic upgrade head`
against the empty database, or use `alembic stamp head`.

Subsequent revisions should provide a meaningful `downgrade()` â€” the
project linter (`ruff` + `pre-commit`) will not flag missing downgrades
but reviewers should require them.

## CI integration

CI applies migrations against an ephemeral Postgres in PR builds:

```bash
docker compose -f docker-compose.ci.yml up -d postgres
TOKENDNA_PG_DSN="postgresql://ci:ci@localhost:5432/ci" alembic upgrade head
pytest -m pg_integration
```

The `pg_integration` mark is reserved for tests that need a live
Postgres; these are not part of the default `pytest` run.

## Troubleshooting

| Symptom | Likely cause | Fix |
|---------|--------------|-----|
| `ModuleNotFoundError: alembic` | `alembic` not installed | `pip install alembic` (it's in `requirements-prod.txt`). |
| `Can't load plugin: sqlalchemy.dialects:postgresql` | `psycopg` not installed | `pip install "psycopg[binary]" psycopg-pool`. |
| Baseline reports "module failed" | A target module raised on import | The migration logs the offending module + exception. Fix the import or remove the module from `_INIT_MODULES` if it has been retired. |
| `target database is not up to date` | Skipped revisions | `alembic current` to see where you are; `alembic upgrade <rev>` to step forward. |
