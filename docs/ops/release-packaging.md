# Release Packaging

TokenDNA should ship as a local appliance bundle plus Python packages for the SDK, collector, and platform integration layer. The goal is for a customer to deploy the control plane inside their own network, next to their identity systems and AI agent runtimes.

## Artifacts

Each enterprise release should include:

- Signed control-plane container image.
- `docker-compose.yml` and `docker-compose.production.yml` for local appliance deployment.
- TokenDNA SDK wheel.
- TokenDNA collector wheel or container image.
- TokenDNA platform package for local ingestion, engines, alerts, and response routing.
- SBOM and provenance metadata.
- Release manifest generated by `scripts/build_release_bundle.py`.
- Local appliance and backup runbooks.

## Build Manifest

Generate the operator manifest:

```bash
python scripts/build_release_bundle.py --image-tag tokendna/control-plane:<version>
```

The default output is:

```text
dist/release-bundle/tokendna-release-manifest.json
```

The manifest records the product version, git SHA, deployment model, package paths, runbook hashes, required gates, and SBOM pointer.

## Required Release Gates

Before signing a release, run:

```bash
ruff check .
python -m pytest
python scripts/build_release_bundle.py --image-tag tokendna/control-plane:<version>
docker compose -f docker-compose.yml -f docker-compose.production.yml run --rm tokendna-deployment-gate
```

The Docker gate verifies production preflight, applies TokenDNA storage migrations, and runs the live Postgres smoke path.

## Customer Deployment Model

TokenDNA is not packaged as a hosted service. A pip install is appropriate for the SDK, collector, and local platform components, but the control plane itself should run as a signed appliance image with Postgres, Redis, and ClickHouse colocated or connected through customer-owned infrastructure.

This keeps identity telemetry, audit state, agent workflow evidence, and enforcement policy inside the customer boundary.
